# SeaChat 投资研究工作台产品规约

## 1. 产品定位

SeaChat 是一个面向个人投资者和轻量专业研究者的投资研究工作台。第一版本不做交易、不做荐股、不替用户下结论，而是把行情、估值、情景假设和风险检查压缩到一个可操作的分析流程里。

优先级：

1. 专业投资研究工作台：提供严肃、可复查的估值分析流程。
2. 普通用户股票体检入口：让非专业用户能快速理解一只股票贵不贵、风险在哪。
3. 极致估值计算器：DCF、倍数估值、安全边际和情景比较必须清晰、快速、可调。

## 2. 目标用户

### 2.1 核心用户

有基本投资知识、会看财报、希望快速建立估值区间的个人投资者。

典型行为：

- 搜索股票或指数。
- 查看当前估值分位、行情、关键倍数。
- 调整收入、自由现金流、增长率、折现率、安全边际等假设。
- 比较保守、基准、乐观三种情景。
- 得到估值区间、风险提示和可复查的模型输出。

### 2.2 次级用户

普通投资者或学习者，只想知道一只股票当前是否处于高估、合理、低估区间。

典型行为：

- 输入股票代码或名称。
- 查看一页式体检结论。
- 理解为什么系统给出这个区间。
- 在需要时展开高级假设。

## 3. 非目标

本阶段明确不做：

- 不做自动交易。
- 不做买卖建议。
- 不承诺实时行情作为交易级数据源。
- 不做完整组合管理。
- 不做社交、评论、内容社区。
- 不做用户账户系统，除非后续需要保存研究案例。
- 不接入需要付费密钥或合规审核的金融数据源。

## 4. 核心工作流

### 4.1 快速体检流

入口目标：让用户在 30 秒内得到一只资产的初步判断。

步骤：

1. 搜索股票、指数或行业，支持代码、名称、拼音和模糊匹配。
2. 系统拉取行情、PE/PB/PS、估值历史分位。
3. 页面显示当前状态：低估值、正常区、高估值或数据不足。
4. 用户看到价格、涨跌幅、主要倍数、历史分位、数据更新时间。
5. 用户可以进入“高级估值”继续建模。

验收标准：

- 股票搜索结果可用。
- 搜索必须支持模糊搜索，输入不完整名称、拼音片段或代码片段时也能返回候选。
- 搜索必须有最近使用、热门资产或本地历史候选，避免空白输入状态。
- 报价失败时有明确错误状态。
- 估值历史不支持时不阻塞主流程。
- 页面不能让用户误以为系统给出交易建议。

### 4.2 专业估值流

入口目标：让用户可控地建立估值区间，而不是只看一个数字。

步骤：

1. 用户选择资产。
2. 系统自动填充可获得的行情和倍数数据。
3. 用户输入或校准财务假设：收入、自由现金流、EPS、股本、净现金。
4. 用户调整模型假设：增长年限、增长率、折现率、永续增长率、安全边际。
5. 系统输出 DCF 估值、倍数估值、综合公允价值、买入观察价、当前价格位置。
6. 系统展示保守、基准、乐观三种情景。
7. 用户可看到模型敏感项：增长率、折现率、永续增长率。

验收标准：

- 所有关键假设都可编辑。
- 每次输入变化都即时刷新结果。
- 永续增长率必须低于折现率。
- 估值结果必须显示计算依据，而不是只给结论。

### 4.3 学习解释流

入口目标：降低理解门槛，但不牺牲专业用户效率。

步骤：

1. 默认只展示关键结论和核心输入。
2. 用户展开解释区，看到 DCF、PE/PB/PS、安全边际的说明。
3. 解释内容与当前模型输入相关联。

验收标准：

- 解释内容不占据专业工作台主视觉。
- 不使用营销式空话。
- 每个解释都能帮助用户理解一个具体输入或输出。

## 5. 信息架构

### 5.1 顶层区域

采用移动端 first 的工作台布局。手机上优先完成完整分析闭环，桌面端再扩展为多栏工作台。

核心区域：

- 顶部全局搜索与当前资产状态。
- 首屏资产体检摘要。
- 估值结论与安全边际。
- 假设调节面板。
- 情景对比。
- 历史估值图表。
- 风险与解释。

桌面端增强：

- 左侧资产与观察列表。
- 中央估值建模区。
- 右侧结论、风险和情景摘要。
- 底部或次级区域展示历史估值图表与学习解释。

### 5.2 页面模块

必须模块：

- Asset Search：资产搜索、类型过滤、快速选择。
- Fuzzy Search：代码、名称、拼音、简称、历史搜索的模糊搜索。
- Market Snapshot：价格、涨跌幅、成交额、市值、PE/PB/PS。
- Valuation Health：历史分位、估值状态、更新时间。
- Assumption Editor：财务与估值假设输入。
- Valuation Output：DCF、倍数估值、综合估值、安全边际。
- Scenario Compare：保守、基准、乐观。
- Local Persistence：本地保存最近资产、用户输入假设、UI 状态和草稿。
- Risk Notes：模型限制、数据缺失、异常输入提示。

可选模块：

- Watchlist：观察列表。
- Saved Cases：保存研究案例。
- Export：导出为图片或 Markdown。

## 6. 数据约束

当前数据来源：

- 东方财富搜索接口。
- 东方财富行情接口。
- 东方财富估值历史接口。
- 用户手工输入的财务假设。

约束：

- 报价缓存时间当前为 8 秒。
- 列表缓存时间当前为 60 秒。
- 历史估值缓存时间当前为 3600 秒。
- 外部接口失败必须优雅降级。
- 财务数据不自动拉取，用户需要手动校准。
- 用户输入、最近搜索、观察列表、估值草稿需要本地持久化。

### 6.1 本地持久化

优先方案：

1. IndexedDB：保存估值案例、观察列表、最近搜索、每只资产的假设草稿。
2. localStorage：只保存轻量 UI 偏好，例如最近打开的 tab、折叠状态、主题偏好。

约束：

- 不依赖登录即可恢复用户上次分析状态。
- 本地数据必须可清空。
- 版本升级需要兼容旧结构，不能让旧数据导致页面崩溃。
- 隐私提示必须清楚说明：数据保存在当前浏览器本地。

### 6.2 搜索体验

搜索必须支持：

- 股票代码精确搜索。
- 股票名称模糊搜索。
- 拼音和拼音首字母搜索。
- 最近搜索优先展示。
- 无结果状态推荐热门资产或提示换关键词。
- 网络失败时展示本地最近搜索，不让搜索面板空白。

搜索交互要求：

- 输入后 150-300ms debounce。
- 候选项显示名称、代码、市场、类型和关键估值指标。
- 点击候选后立即进入资产体检，不要求二次确认。
- 移动端搜索框必须是首屏核心入口。

## 7. UI/UX 方向

视觉目标：

- 专业、冷静、密度高。
- 重点数据一眼可扫。
- 图表和数字是主角，不做营销型大 hero。
- 动效服务于状态变化、层级切换和数据反馈。
- 移动端优先，核心操作必须适合单手使用。
- 搜索、调参和结论查看不能依赖 hover。

推荐 UI 来源：

1. Tremor：核心数据面板、图表、指标卡。
2. shadcn-admin：工作台布局、表单、导航、设置面板。
3. Magic UI：搜索入口、状态切换、关键结论的局部动效。

不建议方向：

- 纯 landing page 模板。
- 大面积玻璃拟态和装饰性渐变。
- 过度卡片堆叠。
- 只追求炫酷但降低数据扫描效率的动效。

## 8. 第一版本验收标准

V1 完成条件：

- 用户能搜索资产并看到行情快照。
- 搜索支持模糊匹配、最近搜索和网络失败降级。
- 用户能完成一次 DCF + 倍数估值。
- 用户能看到历史估值分位或明确的数据不可用状态。
- 用户能比较三种情景。
- 用户刷新页面后可以恢复最近资产和未完成的估值输入。
- 页面以移动端 first 实现，在桌面端增强为专业工作台。
- Cloudflare Pages 可构建和部署。
- 没有控制台运行时错误。
